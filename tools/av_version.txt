# Encoder/Player build version for YouTube titles: format YYYYMMDD.eN.pM
#
# - e = encoder version. Bump when the encode pipeline (sim.py params/code OR
#   pack_stream.py params/code) changes in a way that alters the output.
# - p = player version. Bump when the on-disc player (boot/movieplay_*.s) changes.
# - Bump one or both as appropriate; never decrease either.
# - When bumping, if today's date differs from `date` below, set `date` to today.
# - This is the YouTube-title build version only. The MOVIE.DAT header version
#   field is separate and must NOT be touched.
#
# The title version string = <date>.e<e>.p<p>, e.g. 20260710.e2.p2
#
# History:
#   e1 : Flbk improve-mode default; baseline encoder.
#   e2 : streaming ring fixes — TANK below RING_SIZE (jitter margin), pack
#        forward-fill (use all 5 sectors, prefetch the ring reserve), RING_CAP
#        tied to the player's real RING_SIZE. Enables full-screen H40 (1120-cell).
#   e3 : full-screen H40 desync fix — root cause = back-pressure ratchet. pack
#        RING_CAP lowered well below the player's pump_poll back-pressure
#        threshold (416KB) via CBRSIM_RING_CAP_KB=300 so pump never stalls (no
#        CDC sector drop). Plus per-frame cold cap CBRSIM_PACK_MAXCOLD. Full ED
#        now completes clean on hardware (GPGX).
#   e4 : full-screen H40 quality+stability — RING_CAP raised to 350KB (still below
#        the 416KB back-pressure threshold) so the ring holds a larger jitter
#        buffer, letting the per-frame cold cap rise to 260 (from 192). Verified:
#        the deterministic ring model can NOT predict the collapse (collapse
#        frames show full occ); RING_CAP is the empirical jitter-margin lever.
#        Much higher tile-update density; full ED completes clean on hardware.
#   e5 : per-frame sync marker (frame_seq at the head of each control block) so
#        the player can detect stream desync. Enables robust slip recovery.
#   e6 : audio fix — the PCM WAV is pcm_u8 (unsigned, center 128) but the pack
#        had converted it as signed 8-bit AND fed the WAV header in as samples,
#        scrambling the waveform (~15% of samples flipped). Now strips the header
#        (wave module) and converts u8(center 128) -> RF5C164 sign-magnitude
#        correctly (silence 128 -> 0x00). Audio garble gone.
#   p1 : baseline PCM player.
#   p2 : graceful ring underrun (drained scene shows stale/residual, not garbage).
#   p3 : full-screen H40 completes clean. Fixes: frame0 header loaded via staged
#        CD->Word-RAM->copy (direct CDC_TRN slips); pump during frame0 decode
#        (CDC-overflow drop desynced frame1+); f0pat in Word-RAM scratch (no ring
#        hole); per-frame frame_seq check (detect desync); CD-slip recovery = on
#        MSF gap, re-seek and re-read the lost sector (exact, no quality loss);
#        debug HUD S=slip/re-seek count, D=desync count.
#   p4 : audio glitch fix (the earlier FD-follow / big-lead attempt was REVERTED —
#        it added growing audio lag — so p4 never shipped until now). Shipped p4 =
#        SYNC_MIN lowered (0xC00->0x400): the audio write/play lead may dip briefly
#        at a heavy scene-cut cluster WITHOUT triggering a re-sync. The old re-sync
#        jumped the write ahead and replayed stale loud data = an audible glitch
#        (machi_op F1056). Root-caused with an on-HUD instrument (R=audio re-sync
#        count, L=current lead): the lead troughs ~1467, staying above SYNC_MIN
#        (1024) and overtake (0), so R stays 1 (no re-sync) and no play-overtake.
#        Glitch gone at FULL quality (uncapped) with no A/V lag.
#   e7 : streaming jitter margin — pack RING_CAP raised 350->380 (more prebuffer
#        for the heavy scene-cut cluster; still below the 416KB back-pressure
#        threshold, under=0). Fewer CD-sector drops (38->32) with no quality
#        change (same decisions/cold cap).
#   e8 : clean stream (zero CD-sector drops) — pack per-frame cold cap
#        (CBRSIM_PACK_MAXCOLD=200, frame0 excluded so it stays a full load) caps
#        new-tile load at heavy scene-cut frames so the Sub never overruns the
#        CDC. S(slip)=0 all the way, steady 15fps, so the fixed-rate PCM stays
#        A/V-synced (no re-seek pauses, no rewind, no lag). Capped cells hold the
#        previous frame 1 frame (residual). Resolution/audio unchanged.
#   e9 : cap moved into the encoder + one source of truth. The cold cap now
#        lives only in the sim (CBRSIM_MAX_COLD=185, smart reuse), NOT the pack,
#        so the analysis (from the sim) matches the disc exactly (frame-aligned
#        corr 1.000; the old pack cap dropped bottom cells stale, corr 0.814).
#        av_config.py is the single source for ring (RING_SIZE->RING_CAP/TANK,
#        was drifted 420/380/440) and the drop-safe realized-cold ceiling; the
#        pack ASSERTS realized cold <= ceiling and refuses a pack-side cap; the
#        build asserts the player RING_SIZE matches. Result: S=0, analysis==disc.
#   p5 : Sub-CPU throughput -> higher cold ceiling (better quality). The expand loop
#        called pump_poll every bitmap byte (~140x/frame on full-screen), each a 15-reg
#        save + BIOS_CDC_STAT; that fixed overhead made heavy scene-cut frames overrun
#        the CD (sector slip) above realized cold ~200, forcing a low cap (quality loss).
#        Now pump_poll runs every 8 bytes (~18x/frame, still ample: 1 CD sector per ~166k
#        cycles). Measured on full-screen machi_ed: slips vanish up to realized 429 (S=0),
#        so the cap rises from 195 to 362 (sim cap 350), lifting mean cold 167->230 (+37%)
#        = the Tank/CD-bandwidth is now spent on fresh tiles instead of wasted. Verified
#        S=0, R=1, D=0. Also improves machi_op (faster Sub = shallower audio-lead dips).
#   p6 : debug HUD reposition — the on-screen readout (F/P/S/D/R/L) is now a single
#        row in the top-left corner with 1-tile gaps (was 2 rows, indented), so it
#        covers less picture and every value reads on one line. Positions are named
#        constants (HUD_ROW/HUD_PITCH/HUD_COL_* in movieplay_ip.s), kept in sync with
#        the capture OCR (tools/read_frameno.py: read_hud). DEBUG-only overlay; the
#        shipped (non-debug) codec/stream output is unchanged.
#   e10: CRAM pre-load (MOVIE.DAT v3). All segment palettes ship once in a PALTAB
#        region after the header; the per-frame control block carries only a 1-byte
#        segment reference (pal = seg+1) instead of a 128-byte in-stream CRAM payload.
#        pack_stream.py writes the PALTAB + paltab_sec header field and drops the
#        in-stream CRAM; sim.py stops deducting the 128-byte palette write from the
#        switch frame's budget. The rendered pixels are byte-identical to e9 (verified:
#        machi_op + machi_ed decode_verify diff=0), so this is a delivery/robustness
#        change, not a look change. Payoff: palettes no longer depend on stream timing
#        (a CD slip/re-seek can't corrupt a segment's colours) and the heavy scene-cut
#        switch frame's byte budget is freed — the base for adding more switch points.
#   p7 : player side of CRAM pre-load. At boot the Sub stages the PALTAB into Word-RAM
#        and the Main CPU copies it once into a Main-RAM table (PALTAB_MAX_SEG=64, 8 KB);
#        bf_flip now indexes that table by the 1-byte segment ref instead of reading a
#        per-frame in-stream CRAM block (CPU CRAM write, same as before — only the source
#        moves to Main-RAM). av_config.PALTAB_MAX_SEG is the single source, build-asserted
#        against the player equ. Verified on emulator: full machi_ed plays S=0 D=0 with all
#        14 segments (P0->P0D) switching to correct colours.
#   e11: CRAM-switch trigger generalized from "deep black" to "uniform screen" (any flat
#        colour). A deep blackout is just one case of the screen being one flat colour, so
#        tile reuse is maximal and a CRAM swap is cheap and its colour pop hidden; a
#        white-out or any flat-colour fade is an equally cheap switch point. sim.py now
#        detects uniform frames (fraction of pixels near the frame's mean colour) as a
#        separate, independent clustering and merges those boundaries in (so it never
#        bridges/merges the existing dark clusters). machi_op gains 3 switch points it
#        missed before (two white-outs + a flat stretch, deep mid-segment): 7 -> 10
#        switches, each giving 48-83% quantization-error reduction on the scene it opens.
#        machi_ed is unchanged (its long segment 12 has no uniform moment; that needs the
#        hard-cut path — issue #14, Phase 2). Env: CBRSIM_SEG_UNIFORM / _TOL / _NEAR.
#   e12: MOVIE.DAT v4(可変フレーム)。5セクタ固定padding廃止(各frame=n_pay+n_ctrl sec)。AUDIO
#        をfps由来(15=887,30=443)に=旧887固定が30fpsでdiscをCD 1x 8%超過させていた主因。
#        cold cap基準を15fps=350に(cap(fps)=350*15/fps, 30->175)。真の30fps(N2=29.97)がCD 1x内。
#   p8 : player v4対応。ヘッダのN(vsync/コマ)/AUDIO読込、pump1を可変フレーム化(0セクタは
#        read無しでスキップ, (n_pay+n_ctrl)で前進)、音声可変、stream_totalはファイルサイズ。
#        Sonic H32 30fpsは実機でD=0完走するが Sub pump律速で実効~16fps(スリップ多発)=issue#15待ち。
#   e13: v4レートマッチpadding。各コマを CD 1x(75 sec/s)の1コマ配送レート(=75/fps セクタ)まで
#        padding(累積器で整数割当: 15fps→5固定=v3と同一, 30fps→2/3平均)。paddingを外したv4は
#        ディスクが表示より速く読まれ過剰配送→バッファ溢れ→CDCスリップ(ed実機S=689)していた。
#        レートマッチで「ディスク読み速度=表示速度」に一致=スリップ根絶。header offset56=名目fps。
#   p9 : player レートマッチ対応。pump1が fsec=max(n_pay+n_ctrl, ratedelta) セクタで1コマ前進
#        (padセクタは読んで捨て、累積器sec_accでpackと同期)。h_fps_int@56読込。ip.sのvsync明示
#        ペーシングは撤去(ディスクがレートマッチ=旧来のデータ律速で正しいfps)。ed実機 S=0 D=0
#        R=1 15.0fps=v3同等を確認。30fpsも同枠組みで過剰配送ぶんのスリップが減る見込み。
