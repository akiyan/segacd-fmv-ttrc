# record-mcd: Sega CD Playback Recording Skill

Record a built disc with RetroArch + Genesis Plus GX headless mode using
RetroArch's built-in FFmpeg recorder. Trim the BIOS / CD player / START lead-in
and produce a shareable MP4.

The recording must be the emulator's own synchronized A/V output: video plus
the audio generated by that build. It is not an offline replacement with
`PROBE.BIN` or any other extracted audio, unless the user explicitly asks for
that.

Run from the repository root.

## Preconditions

Required tools:

- `retroarch`
- Genesis Plus GX libretro core
- `Xvfb`
- `xdotool`
- ImageMagick `import` and `montage`
- `ffmpeg` / `ffprobe`
- `python3`

The Sega CD BIOS must be available in RetroArch's system directory. Environment
overrides:

```sh
CORE=/path/to/genesis_plus_gx_libretro.so
SYSTEM_DIR=/path/to/retroarch/system
OUTDIR=/home/akiyan/segacd-novel/videos
```

## Standard Recording Procedure for Agents

```sh
tools/record_movie.sh [--disc CUE] [--out MP4] [--seconds N] [--trim SEC|--auto-audio-trim] \
                      [--tag NAME] [--display :N] [--preset NAME] \
                      [--record-size WxH] [--no-build]
```

Before recording, run the shared-machine exclusion check from `AGENTS.md` and
wait if a sim/render or another emulator is active. Never run two captures.

Defaults and conventions:

- Current main disc: `--disc out/MOVIEPLAY.cue`.
- Older scripts may default to `out/SCFMV_MCD.cue`; override the disc when
  recording the current player.
- Use a realtime preset for synchronized A/V.
- `tools/run_headless.sh` applies the SDL audio clock to **every** recording,
  including `ffv1-flac`, and rejects a duration outside 0.60x--1.50x of the
  requested wall-clock run. Do not bypass this guard with a direct RetroArch
  command.
- **The capture is RetroArch's own lossless recording; Xvfb only provides the
  headless X display and does NOT record.** RetroArch's built-in FFmpeg recorder
  writes the raw mkv at `tmp/<tag>.mkv`:
  - Default (`--preset realtime` → `flac-fast`): x264 **CRF 0** video + **FLAC**
    audio = lossless. This is the stable default used for verification/audio
    recordings this project relies on.
  - `--preset ffv1-flac`: FFV1 video + FLAC = fully lossless including chroma.
    Use this **when pixel-exact dither matters** and analyze that mkv, not the
    transcoded h264 mp4 — a 4:2:0 transcode blurs the dither.
- Output goes under `videos/` (git-ignored), not `tmp/`. When the recording
  corresponds to a sim encode, name it `videos/<stem>_emu.mp4` (same
  `<stem> = <input-basename>_<mode>_<resolution>_<audio>` as the sim run; see
  AGENTS.md "Output Paths"). Sidecar files (audio wav/json, raw mkv, logs) land
  beside it.
- Use a tag such as `rec_<disc_name>` (or `<stem>_emu`).
- Pick an unused X display, for example `--display :236`.
- Default trim is automatic: the tool scans the recorded WAV and chooses the
  loudest requested window, which avoids producing a silent MP4 from a quiet
  boot or dark segment.
- Use fixed `--trim SEC` only when the exact timestamp matters. Fixed trim
  disables auto trim.
- If the disc is not built and rebuilding is wanted, omit `--no-build`.
- If the disc is already built, use `--no-build`.
- For a short check use `--seconds 20` or `--seconds 30`.
- For full-length captures, choose a length long enough for the complete movie.
- If the display collides, change `--display :NNN`.
- Capture the final `OUT=...` line and provide that MP4 path to the user.

Examples:

```sh
tools/record_movie.sh --disc out/MOVIEPLAY.cue --no-build --seconds 30 \
  --tag rec_check --display :269 --out videos/<stem>_emu.mp4

tools/record_movie.sh --disc out/MOVIEPLAY.cue --seconds 180 \
  --out videos/<stem>_emu.mp4
```

For a lossless H32 debug/HUD verification, use this exact low-freedom path:

```sh
ps -eo pid,etimes,args | grep -v grep \
  | grep -iE "sim\.py|render_analysis\.py|retroarch|Xvfb|record_movie|run_headless"
make disc DEBUG=1
OUTDIR="$PWD/videos" tools/run_headless.sh out/MOVIEPLAY.cue \
  --tag <stem>_debug --record --record-preset ffv1-flac \
  --record-size 256x224 --audio-min-rms 1 \
  --shots 68 --interval 2 --display :NNN
```

Do not omit audio synchronization options when composing a lower-level command.
The script now supplies them automatically, but the post-run timing check remains
mandatory evidence. For H32, keep the raw core surface at native 256x224. The
player uses the same contiguous 22-cell HUD in every mode:
`FxxxxPxxSxxDxxRxxLxxxx`. F/L have four hexadecimal digits; P/S/D/R have two.
It fits the 32-column H32 plane, while H40 remains native 320x224.

After every verification capture, require all of the following before trusting it:

- `ffprobe`: expected codec, 256x224 for H32 or 320x224 for H40, 60000/1001 fps, audio present.
- `record timing:` is near the requested wall-clock duration; never accept a
  multi-times-faster capture.
- RetroArch log ends with normal core unload and `Average monitor Hz` near 60,
  not about 300, and does not stop at `SET_GEOMETRY`.
- Audio JSON exists with nonzero RMS and zero clip/jump candidates.
- Extract a movie frame and confirm the DEBUG HUD is visible before starting a
  long OCR scan. If it is absent, rebuild with `make disc DEBUG=1`; do not reuse
  a stale release Main-IP object.
- Read HUD fields from full-width frames and require the counters requested by
  the task (normally `S=0`, `D=0`, and `R=0`) for the complete loop.

## Artifacts for `--tag rec_check`

- `videos/rec_check_movie.mp4`: shareable MP4.
- `videos/rec_check_movie_audio.wav`: audio extracted after MP4 transcode.
- `videos/rec_check_movie_audio.json`: post-transcode audio analysis.
- `videos/rec_check.mkv`: raw RetroArch recording.
- `videos/rec_check.wav`: raw recording audio.
- `videos/rec_check_audio.json`: raw audio analysis.
- `videos/retroarch_rec_check.log`: RetroArch log.
- `videos/xvfb_rec_check.log`: Xvfb log.

## Verify an Existing Recording Without Re-recording

```sh
tools/verify_recording.sh tmp/rec_check_movie.mp4 \
  --out-prefix tmp/rec_check_postcheck
# -> tmp/rec_check_postcheck_verify.wav / .json
```

## Headless Smoke Capture Without MP4

```sh
tools/run_headless.sh out/MOVIEPLAY.cue \
  --tag smoke --shots 8 --interval 1 --display :250

tools/run_headless.sh out/MOVIEPLAY.cue \
  --tag raw --record-realtime --shots 16 --interval 1 --display :251
```

`run_headless.sh` starts Xvfb, waits for boot, sends START, captures
screenshots, and stops RetroArch through the network command interface before
falling back to Escape / kill as a last resort. If the MKV has no duration or
`N/A` duration, treat the run as failed.

## Audio Check Rules

The analyzer checks left and right channels independently.

Representative thresholds:

```sh
--audio-jump-threshold 12000   # practical default gate
--audio-jump-threshold 6000    # stricter investigation gate
--audio-min-rms 1              # fail silent output
--no-audio-check               # only while isolating harness failures
```

A healthy mechanical report usually has:

- `jump_candidate_count=0`
- `clip_count=0`
- nonzero `rms`, or at least the requested minimum RMS

The JSON is a detection gate, not a replacement for listening. Only say there
are no audible clicks after a human has listened to the final MP4.

## Boot Stability Rules

- Run in the foreground. Do not launch multiple `record_movie.sh` or
  `run_headless.sh` jobs with `&`; Xvfb and RetroArch contention can produce
  black or empty recordings.
- Realtime boot takes time. Send enough START presses. `run_headless.sh`
  defaults such as boot-wait 5, presses 14, and press-gap 1.0 exist because too
  few presses can leave the BIOS or CD player on screen.
- Verify boot from frames extracted from the recorded MKV with `ffmpeg`.
  Live `import` screenshots are unreliable. In actual movie playback, the
  center plane content changes frame to frame. BIOS / CD player screens are
  mostly static.
- Check processes with `pgrep -x retroarch` and `pgrep -x Xvfb`. `pgrep -f`
  can match the shell command text and produce false positives.

## Strict Rules and Common Traps

- Do not kill RetroArch first. That can break the Matroska trailer and cause
  audio extraction errors such as "File ended prematurely".
- Stop through the harness path: Escape, wait for exit, then kill only as the
  last resort.
- Recording is realtime synchronized. If `--seconds` is too short, the movie is
  cut off.
- Outputs and intermediates belong under `tmp/` and `out/`, both gitignored.
- **The raw lossless mkv can be large.** A full-length full-motion capture runs
  to GBs (a mostly-static BIOS screen compresses tiny, so short checks are cheap).
  Delete `tmp/<tag>.mkv` once you have extracted the wav / clip you need. A
  runaway accumulation of large captures once filled the disk and killed the
  session — keep captures bounded and clean up the mkv promptly.

## Failure Triage

- MP4 missing or broken:
  1. Check `tmp/retroarch_<tag>.log`.
  2. Check `tmp/xvfb_<tag>.log`.
  3. Check `ffprobe` duration for `tmp/<tag>.mkv`.
  4. Retry with another `--display :N`.
- Silent output:
  1. Remove fixed `--trim`, or use auto trim.
  2. Keep `--audio-min-rms 1`.
  3. Verify the final MP4 has an audio stream with `ffprobe`.
- User hears noise but JSON is clean:
  1. Preserve the MP4/MKV.
  2. Run `verify_recording.sh`.
  3. Report `rms`, `peak_abs`, `clip_count`, `jump_candidate_count`, and
     `max_jump`.
  4. If the detector cannot find it, ask for a timestamp.

## Report Format

Include:

- output MP4 path
- duration
- whether an audio stream is present
- JSON report path
- key values: RMS, peak, clip count, jump count
- whether human listening was performed

Provide the MP4 to the user when the environment supports file sending.
